# 6월 미니프로젝트 회고

6월은 거의 한 달 내내 미니프로젝트와 함께 흘러간 시간이었다.

레벨 2에서 진행한 미니프로젝트는 지금까지 해왔던 미션과는 조금 달랐다. 이전 미션들은 구현해야 할 요구사항이 주어지고, 그 요구사항을 어떻게 잘 구현할지 고민하는 방식에 가까웠다. 하지만 이번 미니프로젝트는 **무엇을 만들지부터 팀이 직접 정해야 했다.**

처음에는 이 점이 꽤 낯설었다. 자유롭게 만들 수 있다는 건 재미있지만, 동시에 그만큼 어려운 일이기도 했다. 단순히 써보고 싶은 기술을 고르는 것이 아니라, 계속해서 아래 질문에 답할 수 있어야 했기 때문이다.

> 왜 이 문제를 해결해야 하는가?
> 왜 이 기술을 선택했는가?
> 왜 이 정도 범위의 MVP가 필요한가?

이번 활동에서 가장 많이 떠올렸던 말은 다음 문장이었다.

> 기술은 목적이 아니라 문제 해결을 위한 수단이다.

평소에는 새로운 기술을 써보고 싶다는 마음이 먼저 들 때도 많았는데, 이번에는 기술보다 먼저 문제를 바라봐야 했다.

미니프로젝트는 대략 아래 흐름으로 진행되었다.

```text
문제 정의 → 가설 수립 → 사용자 인터뷰 → MVP 구현 → 사용자 테스트 → 회고
```

단순히 앱 하나를 만드는 활동이라기보다는, 짧은 시간 안에 제품 개발의 전체 과정을 경험해보는 시간에 가까웠다.

우리 팀은 기존 출석 앱의 불편함을 줄이기 위한 방법으로 **NFC 출석 방식**을 제안했다. 처음에는 “NFC를 찍으면 출석이 바로 되니까 편하지 않을까?”라는 생각으로 시작했지만, 프로젝트를 진행할수록 질문은 조금씩 달라졌다.

* 정말 사용자가 불편해하는 것은 무엇일까?
* NFC는 그 문제를 해결하기에 적절한 수단일까?
* 우리가 만든 MVP는 검증하고 싶은 가설을 확인하기에 충분할까?

이번 회고에서는 1주차부터 3주차까지의 과정을 돌아보며, 우리가 어떻게 문제를 정의하고 MVP를 만들어갔는지 정리해보려고 한다.

---

## 1주차 — 문제를 정하고 방향을 맞춰가기

1주차에는 팀을 구성하고, 어떤 문제를 해결할지 정하는 데 많은 시간을 썼다.

미니프로젝트의 첫 번째 과제는 아이디어를 내는 것이 아니라 **문제를 정의하는 것**이었다. 누가, 어떤 상황에서, 어떤 불편을 겪는지 설명할 수 있어야 했고, 그 문제를 해결하기 위해 어떤 기술이나 플랫폼을 선택할 것인지도 함께 정리해야 했다.

처음에는 여러 주제 후보가 있었지만, 우리 팀은 **등하교 체크 방식의 불편함**에 주목했다. 출석 앱은 우테코 생활에서 매일 사용하는 서비스에 가까웠고, 실제로 주변에서도 출석 앱 사용 과정에서 불편을 겪는 이야기를 자주 들을 수 있었다.

기존 출석 앱에서 가장 크게 느껴졌던 문제는 단순히 “출석 버튼을 누르기 귀찮다”는 것이 아니었다.

* 와이파이 연결을 기다려야 한다.
* 연결된 것 같은데 앱에서는 인식하지 못한다.
* 앱을 껐다 켜거나 네트워크를 다시 잡아야 한다.
* 출석이 제대로 되었는지 바로 확신하기 어렵다.

그래서 우리는 이 문제를 단순한 UI 개선이 아니라, 출석 과정에서 사용자가 느끼는 **불확실성과 대기 시간**의 문제로 바라보게 되었다.

1주차에는 주제 후보를 정리하고, 등하교 체크 방식 개선과 관련된 설문을 준비하고, 예상 질문과 발표 자료를 만들었다. 동시에 1주차 보고서 PR을 제출하기 위해 팀 정보, 문제 정의, 솔루션, 기술 선택 근거, MVP 범위, 검증 방법 초안까지 정리했다.

이때까지만 해도 우리가 만들 결과물이 완전히 선명하지는 않았다. NFC를 활용하면 편해질 것 같다는 기대는 있었지만, 그 기대가 실제 사용자에게도 통하는지는 아직 알 수 없었다.

정리하면, 1주차는 **답을 찾은 시기라기보다는 우리가 어떤 질문을 검증해야 하는지 정리한 시간**에 가까웠다.

---

## 2주차 — 인터뷰로 가설을 확인하고 MVP를 만들기 시작하다

2주차에는 본격적으로 사용자 인터뷰와 가설 검증을 진행했다.

수업에서 가장 인상 깊었던 말은 다음 문장이었다.

> 기획은 가정의 모음이다.

우리가 아무리 그럴듯한 기획을 해도, 사용자가 실제로 그렇게 느끼는지 확인하지 않으면 그것은 여전히 가정일 뿐이었다.

우리는 HMW, 즉 How Might We 질문을 활용해 문제를 여러 방향으로 펼쳐보았다.

* 어떻게 하면 출석 과정을 더 빠르고 간단하게 만들 수 있을까?
* 어떻게 하면 출석을 잊지 않게 만들 수 있을까?
* 어떻게 하면 네트워크 상태와 상관없이 안정적으로 출석할 수 있을까?

여러 질문 중 우리 팀은 최종적으로 아래 질문에 집중하기로 했다.

> 어떻게 하면 출석 과정을 더 빠르고 간단하게 만들 수 있을까?

이를 바탕으로 세운 가설은 세 가지였다.

| 가설                          | 확인하고 싶었던 것                |
| --------------------------- | ------------------------- |
| 기존 출석 과정은 번거롭다              | 실제로 사용자가 출석 과정에서 불편을 느끼는가 |
| 번거로움의 핵심은 앱 실행과 와이파이 연결 문제다 | 어떤 지점에서 가장 많이 막히는가        |
| NFC 방식은 출석 절차를 단순하게 만들 수 있다 | NFC가 문제 해결 수단이 될 가능성이 있는가 |

인터뷰 질문을 만들 때도 조심해야 할 점이 많았다. “이런 기능이 있으면 쓰시겠어요?”처럼 미래의 의향을 묻는 질문은 피하고, 실제 과거 경험을 중심으로 물어야 했다.

그래서 질문도 아래처럼 구체적인 경험을 묻는 방향으로 준비했다.

* 최근 출석 앱을 사용하면서 불편했던 적이 있었나요?
* 그때 어떤 상황이었나요?
* 출석이 제대로 되었는지 확인하기 어려웠던 적이 있었나요?
* 출석 앱을 사용하면서 반복적으로 하는 행동이 있었나요?

인터뷰를 진행하면서 사람들이 기존 출석 앱을 불편하게 느끼는 이유가 꽤 구체적이라는 것을 알게 되었다. 단순히 앱이 마음에 들지 않는다는 것이 아니라, 와이파이 연결이 불안정하거나, 앱이 로딩에서 멈추거나, 출석이 제대로 되었는지 바로 확신하기 어렵다는 이야기가 반복해서 나왔다.

이 과정을 통해 우리가 세운 가설 중 일부는 꽤 강하게 지지되었다. 기존 출석 과정은 실제로 번거로웠고, 그 중심에는 앱 실행과 와이파이 연결 문제가 있었다. NFC 방식도 출석 절차를 단순하게 만들어줄 가능성이 있어 보였다.

하지만 NFC 방식이 무조건 더 좋은 것은 아니라는 점도 함께 드러났다.

* 태그 위치까지 이동해야 한다.
* 등교 시간에 사람이 몰릴 수 있다.
* 기존 앱처럼 자리에서 바로 출석할 수 있는 장점이 사라질 수 있다.
* 앱을 켜고 태그해야 한다면 기대보다 편하지 않을 수 있다.

이후에는 인터뷰 결과를 바탕으로 MVP 개발을 시작했다. 처음에는 이상적인 출석 앱을 만들고 싶은 마음도 있었지만, 계속해서 “우리가 지금 검증하려는 것은 무엇인가?”를 되물어야 했다.

그래서 실제 우테코 출석 API와 바로 연동하기보다는, Firebase를 활용해 테스트용 MVP를 만드는 방향으로 정리했다.

### MVP에서 하기로 한 것

* Android에서 NFC 태그 인식
* 출석/퇴실 기록 저장
* 출석과 퇴실 자동 구분
* 사용자가 현재 출석 상태를 확인할 수 있는 흐름

### MVP에서 제외한 것

* iOS 개발
* 실제 우테코 출석 API 연동
* 완전한 백그라운드 자동 처리
* 고도화된 보안 기능

2주차에 가장 많이 배운 것은 MVP의 의미였다.

MVP는 기능을 많이 넣은 작은 서비스가 아니라, **검증하고 싶은 가설을 확인할 수 있을 만큼만 만든 제품**이었다. 사용자 인터뷰를 하다 보면 추가하고 싶은 기능들이 계속 생겼지만, 그때마다 이 기능이 정말 지금의 검증에 필요한지 팀원들과 다시 이야기해야 했다.

이 과정이 쉽지는 않았지만, 덕분에 우리가 만들고 있는 것이 단순한 앱이 아니라 하나의 가설 검증 도구라는 점을 더 분명히 이해하게 되었다.

---

## 3주차 — 실제 사용자에게 배포하고 한계를 발견하다

3주차에는 만든 앱을 실제 사용자에게 배포하고 테스트를 진행했다.

이 주차는 개발 중간 공유를 준비하는 시기이기도 했다. 단순히 “어디까지 만들었는지”를 보여주는 것이 아니라, 지난주에 시도하기로 한 것이 실제로 효과가 있었는지, MVP 범위에서 지금 어디쯤 와 있는지, 우리가 만든 것이 어떤 가설을 검증하려는 것인지 함께 설명해야 했다.

우리는 Android에서 NFC를 인식하는 방식, Firebase에 출석 기록을 저장하는 방식, 출석과 퇴실을 자동으로 구분하는 흐름 등을 구현했다.

처음에는 NFC 태그를 읽고 저장하는 것만 생각했지만, 실제 테스트를 위해서는 사용자가 아래 내용을 바로 알 수 있어야 했다.

* 지금 출석 상태인지
* 태그가 정상적으로 인식되었는지
* 출석으로 처리되었는지
* 퇴실로 처리되었는지
* 오늘 기록이 남아 있는지

6월 8일부터 10일까지 테스터들이 실제 등교와 퇴실 상황에서 앱을 사용했고, 이후 인터뷰를 통해 사용 경험을 정리했다. 이때부터 우리가 머릿속으로 상상했던 사용 흐름과 실제 사용자가 겪는 흐름 사이의 차이가 보이기 시작했다.

### 좋았던 점

많은 테스터들이 NFC 방식이 빠르고 단순하다고 느꼈다. 기존 앱에서 겪던 와이파이 대기, 로딩 실패, 앱 재실행 같은 불편이 줄어들었다는 의견도 있었다.

실제로 5명 중 4명이 기존 앱보다 NFC 방식을 더 선호한다고 응답했다.

### 발견한 한계

하지만 동시에 한계도 많이 드러났다.

앱을 반드시 실행한 뒤 NFC를 찍어야 한다는 점은 기대보다 불편하게 느껴졌다. NFC라고 하면 자연스럽게 “그냥 태그만 찍으면 되는 것”을 기대하는데, 앱을 켜야 한다는 조건이 붙으면서 장점이 일부 줄어들었다.

태그 위치도 중요한 문제였다. 태그가 한 곳에만 있으면 등교 시간에 사람이 몰릴 수 있고, 위치가 눈에 잘 띄지 않으면 찍는 것을 잊을 수 있었다. NFC 자체의 기술보다, 실제 공간에 어떻게 배치할 것인지가 사용자 경험에 큰 영향을 준다는 점이 인상 깊었다.

히스토리 UI에 대한 피드백도 있었다. 출석을 찍은 뒤 성공 여부가 명확하게 보이지 않거나, 확인 버튼을 누른 뒤 홈 화면에서 출석 기록이 사라지는 것처럼 보여 헷갈린다는 의견이 있었다.

이 피드백을 들으며 출석 앱에서 중요한 것은 단순히 기록을 저장하는 것이 아니라, 사용자가 **“내 출석이 정상적으로 처리되었다”고 확신할 수 있게 만드는 것**이라는 생각이 들었다.

3주차에는 개선하면 좋을 기능들도 많이 발견되었다.

* NFC 읽기 설정이 자동으로 해제되는 문제
* 히스토리 UI를 날짜별로 볼 수 있게 만드는 문제
* 재적 점수표를 확인할 수 있는 UI
* 위젯이 있으면 더 편할 것 같다는 의견
* 하교를 찍지 않았을 때 아무 상태도 나오지 않는 문제

하지만 이 모든 것을 한 번에 고칠 수는 없었다. 인터뷰에서 문제가 많이 발견되어도 전체 개편을 하기보다는, 가장 중요한 것부터 일부만 개선해야 했다.

그래서 우선 아래 작업부터 진행했다.

* 사용자의 화면을 구분할 수 있게 하기
* 홈에서 NFC를 찍은 후에도 출석 히스토리가 보이도록 수정하기
* 재적 점수표를 볼 수 있는 UI 추가하기
* 용어 통일하기

3주차는 개인적으로 가장 많이 배운 주차였다.

개발하는 입장에서는 “기록이 저장되니까 성공”이라고 생각할 수 있다. 하지만 사용자 입장에서는 성공 여부가 화면에서 분명히 보이고, 오늘 내 상태를 바로 확인할 수 있어야 진짜 성공으로 느껴진다.

또 AI를 활용하는 방식에 대해서도 생각해볼 수 있었다. 이번 프로젝트에서 AI는 단순히 코드를 대신 작성해주는 도구라기보다는, 선택지를 정리하고, 놓친 엣지 케이스를 발견하고, 의사결정을 검토하는 데 더 많이 활용했다.

예를 들어 Firebase를 어떻게 사용할지, NFC 흐름을 어떻게 구성할지, MVP 범위를 어디까지 잡을지 고민할 때 AI에게 대안을 물어보고, 그중 실제 상황에 맞는 것은 팀에서 다시 판단했다.

이 과정에서 AI가 제안한 것을 그대로 받아들이는 것이 아니라, 우리 프로젝트의 목적과 MVP 범위에 맞는지 판단하는 것이 중요하다는 것도 배웠다.

---

## 이번 프로젝트를 통해 배운 것

3주차까지 미니프로젝트를 진행하면서 가장 크게 배운 것은 세 가지였다.

### 1. 문제를 너무 빨리 해결책으로 바꾸면 안 된다

처음에는 “NFC를 쓰면 편하지 않을까?”라는 생각이 앞섰다. 하지만 인터뷰를 하고 테스트를 진행하면서, 우리가 진짜로 해결하려던 문제는 NFC 도입 자체가 아니라 출석 과정에서 느끼는 불확실성과 번거로움이라는 걸 알게 되었다.

### 2. MVP는 작게 만드는 것이 아니라 정확하게 만드는 것이다

기능을 적게 넣는다고 해서 무조건 좋은 MVP가 되는 것은 아니었다. 반대로 기능이 많다고 해서 더 좋은 검증이 되는 것도 아니었다.

중요한 것은 우리가 확인하고 싶은 가설에 필요한 기능인지 계속해서 판단하는 것이었다.

### 3. 사용자 경험은 화면 안에서만 만들어지지 않는다

NFC 태그의 위치, 등교 시간의 혼잡도, 와이파이 연결 상태, 사용자가 앱을 켜야 하는지 여부, 성공 피드백의 명확성까지 모두 사용자 경험에 포함되었다.

앱 안의 UI만 잘 만든다고 해결되는 문제가 아니었다.

---

이번 미니프로젝트는 아직 끝나지 않았지만, 3주차까지의 과정만으로도 꽤 많은 것을 배웠다.

예전에는 기능을 구현하면 어느 정도 끝났다고 생각했던 것 같다. 그런데 이번에는 기능을 만들고 난 뒤부터 진짜 고민이 시작되었다. 사용자가 어떻게 받아들이는지, 어떤 상황에서 불편함을 느끼는지, 운영 환경에서는 어떤 제약이 생기는지 계속해서 확인해야 했다.

그래서 이번 미니프로젝트는 단순히 NFC 출석 앱을 만든 경험이라기보다는, 하나의 아이디어가 실제 서비스가 되기 위해 얼마나 많은 질문을 통과해야 하는지 배운 시간이었다.

아직 완성된 답을 찾은 것은 아니지만, 적어도 이제는 질문을 조금 더 잘 던질 수 있게 된 것 같다.
